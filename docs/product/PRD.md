# JadeAI Career 产品需求文档

状态：Implementation in progress
版本：0.2
日期：2026-07-17

## 1. 背景

当前 JadeAI 已具备简历模板、可视化编辑、PDF/DOCX 导出、AI 对话、JD 分析和模拟
面试，但 AI 修改简历依赖模型正确触发 Tool Calling。当前实现允许 AI Tool 直接写入
数据库，缺少修改预览、版本快照、证据追踪和稳定的降级协议。账号体系也仅覆盖 Google
OAuth 或浏览器指纹，没有数据库账号注册、密码登录和后台用户管理。

本项目将 JadeAI 二开为一个多用户、证据驱动的 AI 求职工作台。用户能够上传个人资料，
或把经过授权的 GitHub 仓库同步为个人职业知识库，审核事实后生成基准简历或针对招聘
JD 的定向简历。

## 2. 产品目标

1. 提供账号密码登录、注册控制和管理员用户管理。
2. 所有简历、知识库、模型配置和面试数据严格按用户隔离。
3. 每个用户可配置多个 LLM 服务，并为不同功能指定模型。
4. AI 对简历的修改必须可验证、可预览、可选择应用、可撤销。
5. 首期来源接入不依赖 GitHub App：文件/目录上传为必备入口，公共 GitHub URL 和
   Fine-grained PAT 为 MVP 远程入口；GitHub App 保留为可选高级模式。
6. 将 `MyUnityResume` 的结构化证据模型落地为首个专用导入器。
7. 支持文字、PDF、DOCX 和图片形式的 JD，并生成不覆盖基准简历的定向版本。
8. 保留并增强 JadeAI 的模板、PDF/DOCX 导出和模拟面试能力。

## 3. 明确不在范围内

- 微信或 QQ 登录、聊天记录或社交数据接入。
- `opentalking` 及语音、数字人、TTS、STT、WebRTC 能力。
- 自动向用户 GitHub 仓库写文件或推送 Commit。
- 账单、充值、额度交易或公共模型网关。
- 未经用户确认，把 AI 建议直接覆盖正式简历。
- 首期建设独立向量数据库。结构化事实和关系检索优先。

上游仓库中已有的微信登录研究文档不会作为本次需求依据，也不会进入实施计划。

## 4. 目标用户

### 4.1 普通用户

- 管理账号、个人资料和模型配置。
- 上传个人资料，或连接 GitHub 并选择允许读取的仓库。
- 审核仓库解析出的职业事实和证据。
- 创建、编辑、复制、版本化和导出简历。
- 上传或粘贴 JD，生成定向简历。
- 基于简历和 JD 进行模拟面试。

### 4.2 管理员

- 控制是否开放注册及邀请注册。
- 创建、禁用、恢复和检索用户。
- 查看安全审计和同步任务状态。
- 配置允许访问的 LLM 网络范围及系统级限制。
- 管理数据迁移和系统运行参数，但不能默认读取用户明文密钥。

## 5. 核心用户流程

### 5.1 首次使用

1. 用户通过邀请或开放注册创建账号。
2. 登录后配置至少一个 LLM 档案并完成连通性测试。
3. 创建空白简历、导入已有简历、上传个人信息文件，或连接 GitHub。
4. 审核系统提取的职业事实。
5. 从已审核事实生成一份基准简历。

### 5.2 个人信息来源与职业知识库

| 模式 | 凭证 | 私有仓库 | 更新方式 | 产品级别 |
|---|---|---:|---|---|
| 文件或目录上传 | 无 | 不适用 | 用户重新上传 | MVP 必备 |
| 公共 GitHub URL | 无 | 否 | 手动检查 | MVP 必备 |
| Fine-grained PAT | 用户 PAT | 是 | 手动、定时轮询 | MVP 必备 |
| GitHub App | App 安装授权 | 是 | 手动、定时、Webhook | 可选高级 |

共同流程：

1. 用户选择来源模式，只提供该模式必需的最小信息。
2. 系统校验路径、文件类型、大小、凭证和远端边界。
3. 系统保存不可变来源快照；上传新内容或远端新 Commit 都创建新版本。
4. 专用或通用解析器生成待审核事实。
5. 用户批准、编辑、拒绝或合并事实。
6. 已批准事实可被多个简历和面试重复使用。

### 5.3 JD 定向简历

1. 用户粘贴 JD，或上传 PDF、DOCX、图片。
2. 系统提取并结构化岗位要求。
3. 系统将要求与已审核事实和基准简历进行匹配。
4. AI 生成 `ResumePatch`，系统展示逐项 Diff、依据和风险。
5. 用户选择应用部分或全部修改。
6. 系统复制基准简历形成定向版本，并保存版本快照。
7. 用户导出 PDF 或 DOCX。

### 5.4 模拟面试

1. 用户选择简历、JD 和面试类型。
2. 系统基于已审核事实追问经历和技术细节。
3. 回答和报告归属当前用户。
4. 报告给出可追踪到简历、JD 和事实的改进建议。

## 6. 功能边界

### 6.1 账号与权限

- 支持用户名和密码登录，邮箱作为可选恢复渠道。
- 注册默认关闭，由管理员开启或使用邀请。
- 普通用户与管理员两种角色。
- 禁用用户后撤销全部活动会话。
- 修改密码后使旧会话失效。
- 管理员不能删除自己，也不能降级最后一个管理员。
- GitHub 来源凭证和 JadeAI 登录身份是两个独立概念；上传和公共 URL 不建立第三方身份。

### 6.2 LLM 档案

- 一个用户可保存多个 Provider/BaseURL/Model/API Key 组合。
- 可按简历生成、JD、图片理解和面试分别绑定默认档案。
- API Key 只在服务端解密使用，浏览器不持久化明文。
- 连通性测试同时检测 JSON、Tool Calling 和 Vision 能力。
- Tool Calling 不可用时必须降级到结构化 JSON 协议。

### 6.3 职业知识库

- 事实类型至少覆盖个人信息、工作经历、项目、技能、教育、证书和成果。
- 每条事实拥有来源、证据位置、状态、置信度和审核记录。
- 导入事实默认是 `draft`，只有 `approved` 事实可用于自动生成正式简历。
- 原始证据和对外简历内容分离，内部风险说明不得进入导出文档。

### 6.4 来源接入

- 未配置任何 `GITHUB_APP_*` 环境变量时，上传、公共 URL 和 PAT 入口仍须可用。
- 上传首期接受带相对路径的 JSON、Markdown、TXT、YAML 文件集合；ZIP 只有在完成流式
  解压和压缩炸弹门禁后才启用。
- 公共 URL 只接受规范化的 `https://github.com/{owner}/{repo}`，不执行任意 URL Clone。
- 私有仓库只接受用户明确创建的 Fine-grained PAT，不接受 Classic PAT。
- PAT 由服务端加密保存，后台任务只携带连接 ID，执行时再解密。
- 上传来源通过重新上传产生新快照；PAT 采用定时轮询；GitHub App 才使用 Webhook。
- 所有来源均不得执行仓库或上传内容中的脚本、Skill、Hook、宏或构建命令。

### 6.5 简历

- 保留现有模板和主题能力。
- 支持 PDF、DOCX、HTML、TXT 和 JSON；PDF、DOCX 为发布门禁。
- 支持基准简历、定向简历及普通副本。
- 每次 AI 应用和关键手工修改均可形成版本。
- 定向简历从基准简历复制，不覆盖基准版本。

## 7. 非功能需求

### 7.1 安全

- 所有受保护仓储查询必须同时包含 `userId` 和资源 ID。
- 密码采用 Argon2id 或经过评审的 scrypt 参数。
- 会话使用 Secure、HttpOnly、SameSite Cookie。
- LLM 和 OAuth 密钥采用带版本的服务端加密。
- GitHub App 模式的 Webhook 必须验签、去重并具备重放保护。
- PAT、GitHub App 私钥和 Webhook Secret 均不得进入响应、日志、任务载荷或 LLM Prompt。
- 上传文件限制类型、大小和解析资源。
- BaseURL 请求必须阻止 SSRF；内网模型仅能由管理员显式放行。
- 仓库和 JD 内容均视为不可信数据，不能改变系统指令。

### 7.2 可恢复性

- 同步、解析和 AI 变更任务必须幂等。
- 数据库迁移必须提供前向验证和回滚说明。
- 变更应用失败时事务回滚，不产生半完成版本。
- 远程连接通过定时对账恢复；GitHub App Webhook 丢失时同样由对账补偿。

### 7.3 可观测性

- 记录登录、管理员操作、密钥配置变更、GitHub 连接、同步、事实审核和简历变更。
- 日志不得包含密码、API Key、OAuth Token 或用户完整私有文档。
- 后台任务需要状态、耗时、重试次数和可读错误原因。

## 8. 产品成功标准

- 默认 E2E 流程“注册或邀请 -> 登录 -> 配置 LLM -> 上传 WorkResume 文件 -> 审核事实 -> 导入
  JD -> 生成并应用定向变更 -> 导出 PDF/DOCX”可从干净数据库重复执行。
- 任意用户不能通过 API、AI Chat、导出或后台任务读取或修改其他用户的数据。
- Tool Calling 不可用的兼容模型仍可生成并应用合法 `ResumePatch`。
- 同一个 GitHub Commit 重复同步不会重复创建来源文档或事实。
- 未配置 GitHub App 时默认发布 Gate 仍能通过；启用 App 时再执行真实安装条件 Gate。
- 公共仓库可无凭证导入；PAT 明文不出现在数据库可读列、API、日志或后台任务参数中。
- 每条 AI 生成的正式简历内容能够追踪到用户输入、已审核事实或明确的人工确认。

## 9. 默认产品决策

- 首个部署形态：单实例自托管。
- 生产数据库：PostgreSQL；SQLite 仅用于轻量本地开发。
- 注册策略：默认关闭，首期邀请注册。
- 来源：上传为必备；公共 URL 和 Fine-grained PAT 为 MVP；GitHub App 为可选高级模式。
- GitHub 同步方向：GitHub 到系统，首期不反向写回。
- 知识事实：先审核再用于正式简历。
- AI 修改：先 Diff 后应用，永不静默覆盖。
- 社交与数字人能力：不在范围内。
